04 Pages & components

  • index.astro — renders the 9 homepage sections (from spec §4) as Astro components. Mostly static; ships near-zero JS.
  • BookingModal — the only interactive island. Opens from any "Book" CTA, runs the 3-step flow (slot → details → pay).
  • admin/index.astro — SSR page guarded by an auth check; renders counters + bookings table.
  • Each homepage component carries its Sanctuary styling from tokens.css + the mockup CSS we already wrote.

05 API routes (server)

RouteDoes
GET /api/availabilityFetch open slots for a session type from Cal.com; return clean list to the modal.
POST /api/create-orderLook up price server-side, create a Razorpay order, return orderId + public key.
POST /api/verify-paymentVerify Razorpay signature → create Cal.com booking → write Firestore booking → return success.
POST /api/cal-webhookOn reschedule/cancel/complete from Cal.com, update the matching Firestore booking.

06 Booking + payment sequence

Key principle: the calendar slot is only booked after payment is verified — no unpaid holds on her calendar.

1. Client opens BookingModal → picks session type 2. Modal GET /api/availability → shows real open slots (Cal.com) 3. Client picks slot + enters details 4. Modal POST /api/create-order → server prices it, returns Razorpay orderId 5. Razorpay checkout opens (public key, client side) 6. On success → POST /api/verify-payment { orderId, paymentId, signature, slot, details } 7. Server: verify HMAC signature with SECRET ├─ invalid → reject (faked success blocked) └─ valid → create Cal.com booking (slot → her Google Calendar) → write Firestore booking { status:'scheduled', calBookingId, ... } → (Cal.com emails client + her) 8. Modal shows warm confirmation
Mentoring CTAs use the same flow with a mentoring session type — or, if you prefer, link those to her Topmate while the counselling flow is the custom one. Default: same custom flow for all.

09 Security & env

  • Secrets in Netlify env, never in repo: RAZORPAY_KEY_SECRET, CALCOM_API_KEY, Firebase service-account JSON, admin email allowlist.
  • Public (browser-safe): PUBLIC_RAZORPAY_KEY_ID, public Firebase web config.
  • Razorpay signature verified server-side only before any booking is written.
  • Prices read server-side from sessions.ts — client can't change the amount.
  • Firestore rules: bookings writable only by the Admin SDK (server); admin reads gated by auth. No public client writes.
  • HTTPS via Netlify; security headers in netlify.toml.
